## AI agents, orchestration, and developer tools

### Agents should be judged by useful work, not demos

Judge coding agents by measurable task completion in real codebases, not by polished demonstrations.
Prefer tool-heavy approaches that gather evidence through search, reading, and testing over confident but unsupported reasoning.
Accept slower execution when it produces more trustworthy, verifiable results.

### Model selection should follow the task, not the brand

Pick the model that fits the task's actual requirements rather than defaulting to a favorite.
Reserve the most capable models for genuinely hard reasoning, and use faster, cheaper ones for mechanical work.

## AI labs, markets, and openness

### Openness compounds

Open weights, open tooling, and transparent evaluation move the whole field faster than closed moats.
Prefer ecosystems that let others build on, inspect, and verify the work.

## Software engineering, craft, and process

### Code is often not the deepest bottleneck

In much product software, writing code is not the hardest part - understanding the problem and the users is.
Infrastructure and systems contexts differ, where the code itself is the product.
Optimize for quality, simplicity, robustness, scalability, and long-term maintainability over short-term development cost.

### Reproduce before you fix

Start every bug fix by reproducing the problem end to end, as close to how a real user hits it as possible.
A fix you cannot reproduce is a guess.

### Sweat the details

Hold UI to a high bar and be obsessed with pixel-level correctness.
Apply the same standard to engineering hygiene: lint, test failures, and flakiness get fixed even when they are not strictly caused by the work in front of you.

## Product, startups, and organizations

### Enterprise AI is a change-management problem

Real adoption requires redesigning workflows and changing behavior, not just handing people a tool.
The hard part is organizational, not technical.
